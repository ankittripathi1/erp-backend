# Repository Guidelines

## Project Structure & Module Organization

This is a Django 5 + DRF backend for the University Management System. Project configuration lives in `config/`, with settings, ASGI/WSGI entry points, and URL routing. Domain code is split into apps under `apps/` such as `core`, `students`, `employees`, `fees`, `accounting`, and `library`. Each app follows the standard Django layout: `models.py`, `admin.py`, `views.py`, `tests.py`, and `migrations/`. The full data model is in `docs/SCHEMA.md`; implementation task notes are in `tasks/`.

## Build, Test, and Development Commands

Create and activate a virtual environment before installing dependencies:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Common commands:

```bash
python manage.py migrate          # apply database migrations
python manage.py seed_initial     # seed idempotent reference data
python manage.py createsuperuser  # create an admin login
python manage.py runserver        # run local server at localhost:8000
python manage.py test             # run Django tests
python manage.py makemigrations   # create migrations after model changes
```

PostgreSQL is required. Defaults are `DB_NAME=ums`, `DB_USER=ums`, `DB_PASSWORD=ums_dev`, and `DB_HOST=localhost`; override them in `.env`.

## Coding Style & Naming Conventions

Use idiomatic Django and Python with 4-space indentation. Keep model fields explicit, use `models.TextChoices` for controlled values, and prefer reusable base behavior from `apps.core.models.TimestampedModel`. App names use snake_case, model classes use PascalCase, fields and functions use snake_case, and migration files should be generated by Django. Keep `apps.core` first in `LOCAL_APPS` because it owns `AUTH_USER_MODEL`.

## Testing Guidelines

Tests currently use Django’s built-in test runner and per-app `tests.py` files. Add focused tests next to the app you modify, naming methods like `test_creates_student_enrollment` or `test_prevents_room_double_booking`. Run the whole suite with `python manage.py test`; for narrower checks, run `python manage.py test apps.students`.

## Commit & Pull Request Guidelines

Git history currently uses concise, imperative commit subjects, for example `Add initial Django project setup with manage.py and requirements.txt`. Follow that style: start with a verb, keep the subject specific, and mention the affected area when useful.

Pull requests should include a short summary, migration notes, test results, and linked task or issue references when available. For model changes, include the generated migration and note any data backfill or seed impact. For API work, mention affected endpoints and schema changes.

## Security & Configuration Tips

Do not commit `.env`, secrets, media uploads, or local database files. Keep development defaults out of production: set a strong `DJANGO_SECRET_KEY`, restrict `DJANGO_ALLOWED_HOSTS`, disable `DJANGO_DEBUG`, and use managed credentials for PostgreSQL.
